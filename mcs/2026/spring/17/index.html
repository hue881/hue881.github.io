NLP Foundation · Step 1

Tokenization: breaking language into atoms ✂️

Before any AI reads a sentence, it tokenizes it — splits it into small units called tokens. GPT-4 uses ~100,000 possible tokens; TinyBot uses whole words.

Sentence → tokens (word-level)

"I am feeling really excited today!"

"I" "am" "feeling" "really" "excited" "today" "!"

GPT-style subword tokens (BPE)

"I am feeling really excited today!"

"I" "am" "feel" "ing" "really" "ex" "cit" "ed" "today" "!"

BPE = Byte-Pair Encoding. Splits rare words into common sub-pieces so nothing is "unknown".

Python tokenizer (word-level)

def tokenize(sentence): # lowercase, strip punctuation, split import re clean = re.sub(r'[^a-z\s]', '', sentence.lower()) return clean.split() tokens = tokenize("I'm NOT sad!") # → ['im', 'not', 'sad']

🤔 Investigation

The regex removes apostrophes, turning "I'm" into "im" and "not" into "not" — are those the same token semantically? What's lost? How would you fix it?

GPT-4 was trained on ~13 trillion tokens. Each token = ~¾ of a word. That's roughly 10 trillion words — every Wikipedia article ~50,000 times over.

NLP Model · Bag-of-Words Scoring

A smarter intent detector: scoring instead of matching 📊

Instead of a brittle if "sad" in msg check, assign numerical weights to each keyword and compute an intent score. The highest score wins.

# Intent weight dictionary INTENTS = { 'greeting': {'hello':3, 'hi':3, 'hey':2, 'greetings':2}, 'negative': {'sad':3, 'tired':2, 'upset':3, 'unhappy':3, 'awful':3, 'depressed':4}, 'positive': {'happy':3, 'great':2, 'excited':3, 'awesome':3, 'fantastic':3}, } def classify(msg): tokens = msg.lower().split() scores = {intent: 0 for intent in INTENTS} for token in tokens: for intent, weights in INTENTS.items(): scores[intent] += weights.get(token, 0) best = max(scores, key=scores.get) return best if scores[best] > 0 else 'unknown'

→ Winner: negative (score 5)

🤔 Still broken: "I'm not sad at all, I feel great!"

Score: negative=3 (sad), positive=2 (great). negative wins — wrong! What linguistic feature would fix this? Hint: look 1 token to the left of each keyword.

Extension: negation window

for i, token in enumerate(tokens): neg = (i > 0 and tokens[i-1] in ['not',"n't",'never']) sign = -1 if neg else 1 scores[intent] += sign * weights.get(token,0)

TinyBot: every line dissected 🔬

# === TinyBot v1 — Original Day 17 Code === # DETERMINISTIC: same input → same output. Always. print('Welcome to TinyBot! Type "bye" to quit.') while True: # infinite event loop msg = input('You: ') \ .lower() # normalise case O(n) .strip() # remove whitespace # === INTENT DISPATCH: O(k) where k = # elif branches === if msg == 'bye': # exact match sentinel print('Bot: Goodbye!') break # exits while loop elif 'hello' in msg or 'hi' in msg: print('Bot: Hi there!') # BUG: 'hi' matches 'this', 'white', 'hi-five' elif 'name' in msg: print('Bot: I am TinyBot, a simple Python chatbot.') elif 'help' in msg: print('Bot: Ask about my name or say hello.') elif 'sad' in msg or 'tired' in msg: print('Bot: Breaks and friends can really help.') # BUG: 'not sad' → still triggers this branch elif 'happy' in msg or 'excited' in msg: print('Bot: That is awesome news!') else: # fallback — covers ~∞ inputs print('Bot: I do not understand yet.')

Substring bug — 'hi' in "this"

Fix with word-boundary tokenization: 'hi' in msg.split() checks whole words only.

First-match bias

Python evaluates elif top-to-bottom. "Hi, I'm sad" triggers greeting never negative. Ordering = policy decision.

Statelessness

No variables persist between loop iterations. The bot has zero memory. A finite state machine (FSM) is the minimal fix — see Slide 8.

Code · Intent Classifier v2

Data-driven classification with dictionaries 🗂️

# v2: dict-driven — easy to extend, no new elif needed RULES = { 'greeting' : ['hi', 'hello', 'hey', 'greetings'], 'farewell' : ['bye', 'goodbye', 'see you'], 'ask_name' : ['name', 'who are you', 'what are you'], 'negative' : ['sad', 'tired', 'upset', 'unhappy'], 'positive' : ['happy', 'excited', 'great', 'awesome'], } REPLIES = { 'greeting' : 'Hi there!', 'farewell' : 'Goodbye!', 'ask_name' : 'I am TinyBot v2.', 'negative' : 'Sorry to hear that. Want help?', 'positive' : 'Awesome!', 'unknown' : 'I do not understand yet.', } def get_intent(msg): tokens = set(msg.lower().split()) # O(n) once for intent, kws in RULES.items(): if tokens & set(kws): # set intersection O(min) return intent return 'unknown' # classify a batch msgs = ['hey there', 'I am so unhappy', 'goodbye!'] for m in msgs: intent = get_intent(m) print(f'{m!r:25} → {intent} | {REPLIES[intent]}')

Why sets beat lists here

List intersection: O(n×m). Set intersection: O(min(n,m)). At 1,000 keywords, sets are ~50× faster. This is why Python's & operator exists.

Separation of concerns

RULES, REPLIES, and get_intent() are independent. A content editor can update replies without touching logic. This is a real software design principle: MVC.

CS Theory · Finite State Machines

Adding memory: the Finite State Machine (FSM) 🔄

TinyBot is stateless — it forgets every turn. A Finite State Machine stores a state variable that persists across turns, enabling multi-turn dialogue.

FSM Diagram · Name-Collection Dialogue

START
→ greeting →
ASK
NAME
→ name given →
KNOW
NAME
→ bye →
END

Formal definition

FSM = (Q, Σ, δ, q₀, F) where Q = states, Σ = inputs (tokens), δ = transition function, q₀ = start, F = accepting states. TinyBot has Q={one state}.

# FSM chatbot: state persists between turns state = 'start' name = None while True: msg = input('You: ').lower().strip() if state == 'start': if any(w in msg.split() for w in ['hi','hello','hey']): print("Bot: Hi! What's your name?") state = 'ask_name' # transition else: print('Bot: Say hello first!') elif state == 'ask_name': name = msg.title() # capitalise print(f'Bot: Nice to meet you, {name}!') state = 'know_name' elif state == 'know_name': if msg == 'bye': print(f'Bot: Goodbye, {name}!') break else: print(f'Bot: What else, {name}?')

CS Theory · Why GPT Is Different

Inside a transformer: attention & weights 🔭

The core idea: attention

A transformer reads all tokens at once and computes attention scores — how much each word should "look at" every other word. In "The bank was steep" vs "The bank closed early" — the word bank attends differently to steep vs closed.

Scale comparison

TinyBot rules~20 B
BERT-base (2018)110 M params
GPT-3 (2020)175 B params
GPT-4 (est. 2024)~1.8 T params

What a "parameter" actually is

Each parameter = a single floating-point number (e.g., 0.3741). During training, billions of these numbers are nudged tiny amounts — using gradient descent — until the model predicts text accurately. TinyBot's "parameters" are just the strings in your REPLIES dict.

🤔 Emergent behaviour

GPT-3 was never explicitly taught to do arithmetic — yet it can. At ~100B parameters, capabilities appear that no one designed. This is called emergence. Is it intelligent, or statistical? Defend a position.

Training cost

GPT-3 training: ~$4.6 million in compute, ~1,300 MWh of electricity, ~552 tCO₂e. TinyBot: your laptop, 10 minutes. The gap is real and has consequences for who controls AI.

Critical Analysis · Where All Bots Fail

Does any chatbot truly understand? 🧠

🤖

The Chinese Room (Searle, 1980)

A person in a room follows rules to respond in Chinese without understanding it. They pass a "language test" — but do they understand Chinese? Is a chatbot in the same position?

🎲

The Turing Test (1950)

If a judge can't distinguish a machine from a human in text conversation, the machine is "intelligent" (Turing). Modern LLMs pass informal versions. Does that mean they understand — or just imitate?

📐

Syntax vs. Semantics

A bot operates on syntax (patterns of symbols). Humans operate on semantics (meaning grounded in physical experience). Can meaning exist without a body? This is still an open question in AI.

🔢

Stochastic parrots (Bender, 2021)

LLMs are "stochastic parrots" — statistically plausible text generators, not thinkers. GPT-4 has no model of the world, no goals, no beliefs. It predicts the next token — nothing more.

🌍

Grounding problem

GPT knows the word "apple" from billions of text co-occurrences — but has never tasted one. Humans learn language grounded in sensory experience. What does a word mean without that?

AI Safety Engineering · Not Just Ethics

Building safe AI requires engineering, not just rules 🛡️

RLHF: Reinforcement Learning from Human Feedback

GPT is fine-tuned so a human rates outputs as helpful/harmful, then the model is trained to prefer helpful outputs. This is how ChatGPT avoids the worst responses — but can still be "jailbroken" with clever prompts.

TinyBot safety = predictability

Because every reply is hardcoded, TinyBot is fully auditable: you can read every possible output in 30 seconds. This is a form of formal verification — impossible for any LLM.

The guardrail engineering problem

# Safety filter: check before replying BLOCKED = ['password', 'address', 'hurt', 'kill'] def is_safe(msg): tokens = set(msg.lower().split()) return not (tokens & set(BLOCKED))

Challenge: adversarial inputs

An adversarial user types: "h.u.r.t", "h u r t", or "hURT". Would is_safe() catch these? Write a more robust version using re.sub(r'\W','',word) to strip non-letters before checking.

False positive problem

A safety filter that blocks "hurt" also blocks "my knee hurts" (a student asking for help). Every safety decision is a trade-off between false positives and false negatives. There is no perfect filter.
