SLIDE 3 -- WARM-UP: HOW CAN ONE VARIABLE HOLD MANY CHOICES?
[Duration: ~6 minutes]
--------------------------------------------------------------------------------

[No screens. Display the two warm-up cards -- "Look & Predict" with the
illustrated rock-paper-scissors image, and "Turn & Talk" with the three RPS
hand-sign tokens (rock, paper, scissors).]

"Here's today's big warm-up question, and I want you to really wrestle with
it before we open anything: if our game has three choices -- rock, paper,
scissors -- should we store them in THREE separate variables, like
choice_one, choice_two, choice_three? Or in ONE organized container that
holds all three? Turn to your neighbor, thirty seconds, make your case
either way."

[PAUSE -- 30 seconds of pair talk. Circulate, listen for early reasoning.
Some students will say 'three variables is simpler.' That's a completely
fair guess at this stage -- don't correct it yet, just collect it.]

[Reveal the "Big idea" card at the bottom of the slide.]

[KEY IDEA]:
"Here's the problem with three separate variables that you'll feel in about
five minutes when we start coding: if I want the computer to PICK one of
your three choices at random, how would it do that with three separate
variables sitting in three different places in memory? It would need some
clunky way to roll a die and then chase down the right variable by name.
But if all three choices live together, in ORDER, inside one list, the
computer can just say 'give me a random item from this list' -- one line,
done. Lists help us group related items so our code can grab one, compare
one, or loop through all of them, using their POSITION instead of needing a
brand-new variable name for every single item."

"Now the Turn & Talk: imagine a partner picks a move -- rock, paper, or
scissors -- and you have to say out loud what beats it, as fast as you can.
Which rule is easiest for YOUR brain to remember instantly? And here's the
real question: which one do you think would be EASIEST for a computer to
check quickly?"

[PAUSE -- sample 2-3 pairs. Guide toward: humans remember rules by feel;
computers check rules by comparing exact text, like the string "rock"
against the string "scissors" -- there's no 'feel,' only exact matching.]

[TIER A note: keep this fully concrete -- point at the three physical hand-
sign images, have them physically hold up rock/paper/scissors while
guessing which one beats which, skip the abstract 'why lists' reasoning
until the mini-lesson shows it in code. TIER C stretch: push further --
"If we had FIVE choices instead of three, would the three-separate-
variables plan get worse faster, or about the same? Why?" (It gets much
worse -- five variable names to track and compare by hand versus one list
that scales for free.)]

--------------------------------------------------------------------------------
SLIDE 4 -- MINI-LESSON PART 1: WHAT IS A PYTHON LIST?
[Duration: ~3 minutes]
--------------------------------------------------------------------------------

[Display the two-card slide -- "Real-world picture" (cubby shelf, playlist,
list) on the left, "Code picture" with the moves list and four print()
lines on the right.]

"Let's answer the warm-up question with real code. First, three real-world
pictures, because the idea of a list existed long before Python did. A
cubby shelf holds many folders in one place. A playlist holds many songs,
in order, in one place. A Python list holds many VALUES, in order, in one
variable."

[KEY IDEA -- go slowly, this is the spine of the whole session]:
"Here's the code: moves equals a square bracket, then 'rock' comma 'paper'
comma 'scissors', then a closing square bracket. Those square brackets are
the signal to Python: 'everything inside here is one collection, stored
together, in this exact order, under this one name -- moves.' That's the
entire syntax for a list. You just built a container that holds three
values using ONE variable name, which directly answers the warm-up: we do
NOT need three separate variables. We need one list."

"Now watch what happens when I print pieces of it: print(moves) shows the
whole list at once, brackets and all. But print(moves[0]) shows just ONE
item -- and that square bracket with a number inside it is called an
INDEX. It tells Python exactly which position to reach into."

[PAUSE -- ask: "If moves[0] is rock, what do you predict moves[2] will
be?" Let a few guesses land -- most will correctly guess scissors by
counting positions. Then ask: "Why do you think counting starts at 0
instead of 1?" Let a few guesses land -- don't fully answer yet, Slide 5
answers this properly.]


--------------------------------------------------------------------------------
SLIDE 5 -- MINI-LESSON PART 2: INDEXING -- HOW DOES PYTHON FIND ONE ITEM?
[Duration: ~4 minutes]
--------------------------------------------------------------------------------

[Display the numbered-boxes card -- index 0 / rock, index 1 / paper, index
2 / scissors -- alongside the "Prediction challenge" card.]

"Here's the answer to 'why does counting start at 0': picture your list as
a row of numbered boxes, left to right. The FIRST box is not labeled '1' --
it's labeled '0.' That's not Python being weird for no reason -- it's
because the index actually measures 'how many steps away from the start of
the list is this item.' The very first item is zero steps away from the
start. The second item is one step away. That's WHY index 0 holds the
first item, index 1 holds the second, and index 2 holds the third."

[KEY IDEA]:
"This matters enormously in real code, because it's the single most common
beginner mistake in almost every programming language, not just Python:
assuming the first item is at position 1. If you ever get an answer that's
'off by one' -- too high or too low by exactly one -- indexing starting at
zero is very often the reason. Get comfortable with it now, because you
will use it in nearly every program from here to the end of the course."

[PAUSE -- run the prediction challenge live: "What prints if I run
print(moves[1])?" Let them answer -- paper. -> "What mistake happens if we
ask for an index that isn't there, like moves[5], when our list only has
three items?" Let them guess. Confirm: Python throws an IndexError -- it
refuses to guess, it tells you plainly that position doesn't exist.]

"One more prompt on this slide: a game menu, a class roster, and a score
history can all be stored as lists too -- this isn't a rock-paper-scissors-
only trick, it's a general tool you'll reuse constantly."

[Point to the misconception-check card.]

"Common mix-up to watch for in your own head today: if you catch yourself
saying 'the first item is index 1,' pause, point at the leftmost box, and
say out loud, 'computers begin at zero.' Say it enough times today that it
starts to feel normal instead of backwards."

[TIER A note: keep this fully hands-on -- have them count on fingers
starting at zero, touching each box in order, before touching any code.
TIER C stretch: ask "If a list has 10 items, what is the index of the
LAST item?" (9, not 10 -- reinforces the off-by-one pattern early.)]


--------------------------------------------------------------------------------
SLIDE 6 -- MINI-LESSON PART 3: HOW DOES THE COMPUTER CHOOSE A MOVE?
[Duration: ~4 minutes]
--------------------------------------------------------------------------------

[Display the "New tool: random.choice()" card with the import random code
block, alongside the "Ask before revealing" card with its three bullet
questions and the "Prediction Stop" callout about lizard and spock.]

"Now for the tool that makes today's game actually feel like a GAME instead
of a script that does the same thing every time. Watch this code: import
random -- that line tells Python 'I want to use the random toolbox that's
already built into the language, I don't have to build randomness myself.'
Then: computer_move equals random.choice(moves)."

[KEY IDEA -- this is the payoff of the whole mini-lesson]:
"Think about what just happened. random.choice() reaches INTO our list --
the exact same list we built two slides ago -- and pulls out ONE item, and
we, the programmers, do not know in advance which one it will be. That
unpredictability is exactly what makes rock-paper-scissors fun to play
against a computer instead of boring. And notice: this only works because
our three choices are organized together in a list. random.choice() needs
something to choose FROM -- a single container of options -- it has no way
to reach into three separate, disconnected variables and pick among them."

[PAUSE -- run the three "ask before revealing" questions live, one at a
time: "Could the computer pick something that is NOT in the list?" (No --
random.choice() can only return an item that's actually inside the list you
give it.) "What must be true before random.choice(moves) can work?" (The
list has to already exist, with at least one item in it.) "Why is a list a
perfect partner for random choice?" (Because random.choice needs a
collection to search inside -- a list IS that collection.)]

"Here's the Prediction Stop on this slide, and I want you to really feel
the power of this: if we add 'lizard' and 'spock' to our moves list, making
it five items instead of three, how much of THIS line of code has to
change?" 

[PAUSE -- let them guess. Reveal: zero characters change. random.choice(moves)
stays exactly the same, because it doesn't care how many items are in the
list -- it just looks at whatever is there. That is the real power of
storing things in a list instead of separate variables: your code scales
without you rewriting it.]

[TIER C stretch: "What Python keyword did we need before we could even use
random.choice() -- and why do you think Python makes us ask for that
toolbox by name instead of giving it to us automatically?" (import random --
Python keeps extra tools separate so your basic programs stay small and
fast unless you specifically ask for more.)]

SLIDE 8 -- AI MOMENT: WHY IS A LIST LIKE A TINY DATASET?
[Duration: ~4 minutes]
--------------------------------------------------------------------------------

[Display the AI Moment badge slide -- "Our game list" (three example
cells: rock, paper, scissors) on the left, "Rule-based today" / "Modern AI
later" cards on the right.]

[ROBOT] -- AI MOMENT: A LIST IS ORGANIZED DATA
"Now my favorite part of every day -- how does today's brand-new tool,
the list, connect to AI? Here's the question on the slide: why is a list
like a tiny dataset?"

[KEY IDEA]:
"Look at our three list cells: rock is labeled 'example 1,' paper is
'example 2,' scissors is 'example 3.' That word -- EXAMPLE -- is exactly
the word data scientists use. A dataset, in AI, is a big organized
collection of examples the computer can use. Our list is tiny -- only three
examples -- but it is structurally the SAME idea: organized data, stored in
order, ready for a program to use."

"Here's the honest contrast, and I want you to hold both halves of it at
once. TODAY, our program is rule-based -- WE wrote the rules ourselves:
rock beats scissors, scissors beats paper, paper beats rock. We typed every
single rule by hand, and the computer just checks them. Modern AI, the kind
you'll start touching tomorrow with Teachable Machine, works differently --
it uses much BIGGER datasets, often thousands or millions of examples, and
it LEARNS patterns from those examples instead of us hand-writing every
rule."

[PAUSE -- ask the question on the card directly: "If our list had only ONE
item instead of three, could the game still feel fair?" Let them answer --
obviously not, there'd be no real choice or challenge. -> "What does that
tell us about why AI needs enough examples?" Let a few answer. Guide
toward: just like our tiny 3-item list needs all three moves to make the
game work fairly, a real AI model needs ENOUGH good examples in its dataset
to make fair, accurate predictions. Too few examples, in either case, means
something breaks or feels unfair.]

"Keep that in your pocket for tomorrow: today you built organized data by
hand, three items, and wrote every rule yourself. Tomorrow you'll watch a
computer build its OWN understanding from data we feed it, live, in
real time."

SLIDE 10 -- GUIDED BUILD: BUILD THE GAME, ONE THOUGHT AT A TIME
[Duration: ~20 minutes -- go slowly, narrate every line, test after Step 4]
--------------------------------------------------------------------------------

[Students open the Python Playground NOW. Walk the room -- confirm every
student has playcode.io/python-playground open with a blank editor before
continuing. Buddy-pair Tier A / Grade 1-2 students. This is the last major
guided build before Phase 3 begins -- treat it with real weight; every
student needs this working before Open Build.]

"Screens ON. Everyone open the Python Playground and make sure you're
looking at a totally blank editor. Thumbs up when you're there. [Wait for
all thumbs.]"

"Here's what every one of us will have by the end of this guided build: a
list of three moves, a random computer pick straight from that list, the
player's own move typed in, and a full set of if/elif/else rules that
correctly decide win, lose, or tie. Do exactly what I do, step by step, no
more, no less -- and we will test together right after Step 4, before
anyone adds a single extra feature."

"STEP 1 -- MAKE THE LIST. Type: moves equals bracket, quote rock quote,
comma, quote paper quote, comma, quote scissors quote, close bracket.
[Wait -- check every screen for correct spelling and matching quotes.]"

[PAUSE -- ask: "Why is this better than three separate variables --
choice_one, choice_two, choice_three?" Let them answer using their own
words from the warm-up. Guide toward: one name to remember, one thing
random.choice can search, and it scales to five or ten moves without any
new variable names.]

"STEP 2 -- IMPORT RANDOM AND PICK FOR THE COMPUTER. At the very top of your
file, above the list, type: import random. Then, after your moves list,
type: computer equals random.choice(moves). [Wait -- check every screen;
this line must come AFTER moves is defined, since random.choice needs the
list to already exist.]"

[KEY IDEA]:
"Quick callback to Slide 6: why does 'import random' have to be the very
first line? Because Python doesn't automatically load every tool it knows
how to use -- only the small core toolbox. random.choice lives in an extra
toolbox called random, and 'import random' is literally us asking Python to
go get that toolbox off the shelf before we try to use anything inside it."

"STEP 3 -- GET THE PLAYER'S MOVE. Type: player equals input(open paren,
quote Type rock, paper, or scissors: quote, close paren), then dot lower()
right after the closing parenthesis of input. [Wait -- check every screen.]"

[PAUSE -- ask: "What exact words should the player type for our comparisons
to work later?" Let them answer -- exactly 'rock', 'paper', or 'scissors,'
matching the spelling inside our list exactly. -> "Why did we add .lower()
right after input()?" Let them guess. Guide toward: if a player types
'Rock' with a capital R, Python sees that as a DIFFERENT string than
'rock' -- .lower() forces everything to lowercase so a capital letter can
never accidentally break the game.]

"STEP 4 -- PRINT BOTH CHOICES CLEARLY, THEN TEST. Type: print(quote You
chose: quote, comma, player). Then on the next line: print(quote Computer
chose: quote, comma, computer). Now RUN your code. [PAUSE -- wait for every
screen to actually print both lines before continuing. This is our
mid-build test -- we are NOT waiting until the very end to check our
work.]"

[KEY IDEA -- tie back to good coding habits]:
"Notice what we just did: we printed both choices BEFORE writing any
win/lose logic. That's not filler -- that's a debugging habit. If something
goes wrong later in the if/elif chain, you can look at exactly what player
and computer actually equal, instead of guessing blindly. Good coders test
in small pieces, constantly, not once at the very end -- that's the same
lesson Slide 9's video modeled for us."

"STEP 5 -- COMPARE THE TWO MOVES. Start with the tie case first: type if
player equals equals computer, colon, then indent and type print(quote Tie!
quote). Then add: elif player equals equals quote rock quote and computer
equals equals quote scissors quote, colon, indent, print(quote You win!
quote). Repeat that elif pattern for paper-beats-rock and scissors-beats-
paper. Finish with a plain else, colon, indent, print(quote Computer wins!
quote), which catches every remaining losing case. [Wait -- check every
screen for correct indentation under each condition -- this is the most
common typo point in the whole build.]"

[PAUSE -- prediction pause from the slide: before anyone runs this, have
students whisper to a neighbor: "Can this code ever print 'You win!' when
both choices match?" Let them answer -- no, because the very first if-check
catches the tie case before Python even looks at any elif branch. -> Now
have everyone actually run it and try to force a tie on purpose to prove
it.]

[KEY IDEA -- why order matters in if/elif chains]:
"This is worth sitting with: Python checks conditions from TOP to BOTTOM
and stops at the very first one that's true. That's exactly why we put the
tie check FIRST -- if we'd put it last, a tie could accidentally get
caught by an earlier elif and print the wrong result. Order in an if/elif
chain isn't just style, it changes what the program actually does."

"Quick troubleshooting before Open Build -- 'Stuck? Check this first':
did you spell 'rock', 'paper', and 'scissors' EXACTLY the same way in your
list, your elif conditions, and when you type your test answer? Is your
if/elif/else chain indented consistently -- no mixing tabs and spaces? Did
you actually click Run after your last edit, not just look at the code? At
least nine times out of ten, a rock-paper-scissors bug traces back to one
of those three things."

[SECTION 1 (75 min): this guided build is the SAME for all sections --
protect the full 20 minutes, since it IS the complete, working game. If
time is tight, compress narration on Steps 1-3 (they're mostly review from
Sessions 9 and 12), but never skip the Step 5 prediction pause or the
mid-build test at Step 4.]

SLIDE 12 -- STAND UP AND BECOME THE PROGRAM
[Duration: ~5 minutes]
--------------------------------------------------------------------------------

[Everyone stands. Display the "Human Rock-Paper-Scissors Logic" break card
-- four IF/ELSE rows mapped to physical hand shapes.]

"Stand up, push your chair in a little, give yourself room. For the next
five minutes, YOU are the program, and I am the input. Here's your
if/elif/else chain, but instead of printing text, you're going to move your
body."

"IF I say 'rock' -- make a rock fist. IF I say 'paper' -- open your hand
flat. IF I say 'scissors' -- make a V shape with two fingers. ELSE -- if I
say something that ISN'T one of those three words, like 'banana' -- freeze
completely still, like a confused computer waiting for a valid input."

[Call out a mixed, quick sequence of "rock," "paper," "scissors," and one
or two nonsense words like "banana" or "purple," speeding up gradually.
Watch for students who freeze correctly on the nonsense words -- that IS
the point of this break, not just wiggling energy out.]

[KEY IDEA -- brief, spoken while everyone's still moving]:
"Notice what just happened in your body: you experienced the difference
between a VALID input, which your if/elif chain knows exactly what to do
with, and an INVALID input, which falls all the way through to else and
does something different -- in this case, freezing instead of crashing.
That's exactly what our program needs to do gracefully if a player ever
types something weird like 'Rock!' with a punctuation mark or 'lizard'
before we've added that move."

"Nice work, program. Sit back down -- let's go build the rest of this."

================================================================================
APPENDIX A -- COMMON PITFALLS & HOW TO HANDLE THEM
================================================================================

PITFALL: Student spells "rock", "paper", or "scissors" inconsistently
across the list, the elif conditions, and their typed test input (extra
capital letter, trailing space, missing letter).
  Handle: "This is the #1 bug in today's whole lesson. Have the student
          read each spelling out loud, letter by letter, comparing the
          list, the elif lines, and what they physically typed when
          testing. Remind them: .lower() only fixes CAPITALIZATION, not
          typos or extra spaces."

PITFALL: The tie check (if player == computer) is placed AFTER the win/lose
elif branches instead of first, so a tie occasionally gets misreported.
  Handle: "Slide 10's exact prediction pause. Python checks top to bottom
          and stops at the first true condition -- move the tie check back
          to the very first if, before any elif, so it's always checked
          first."

PITFALL: random.choice(moves) is written BEFORE the moves list is defined,
causing a NameError.
  Handle: "Python reads your file top to bottom, one line at a time -- it
          has to already know what 'moves' is before that line can use it.
          Move your list definition above the random.choice() line."

PITFALL: Forgetting import random at the top, causing random.choice to
fail immediately.
  Handle: "random.choice lives in an extra toolbox Python doesn't load
          automatically. Add 'import random' as literally the very first
          line of the file."

PITFALL: Score variables (Tier B) are reset to 0 inside the round loop
instead of before it, so the score never actually accumulates.
  Handle: "Same rule as Session 6: SET happens ONCE, before any rounds are
          played. CHANGE happens repeatedly, inside the loop, triggered by
          who won that round. If score = 0 is inside the loop, it resets
          every single time and never remembers anything."

PITFALL: Tier C pattern-memory student's "most common move" logic always
returns the same move regardless of what the player actually played.
  Handle: "Have them print the full list of past moves first, and count
          occurrences out loud before writing the comparison code -- debug
          the COUNTING step before trusting the DECISION step built on top
          of it."

PITFALL: Fast Tier-C student finishes score-keeping, pattern memory, AND
the challenge card early and is bored.
  Handle: Point them to the "fast finisher move" -- refactor the repeated
          win-checking into a helper function or a dictionary of rules --
          or invite them to mentor a neighbor through Steps 4-5 of the
          guided build.

PITFALL: Young student (Grade 1-2) overwhelmed trying to add score-keeping
AND extend the list to five moves at once.
  Handle: Narrow the world. "Today you only need Tier A's goal: one clean
          round that correctly says win, lose, or tie, every single time
          you run it. That's a complete, real, working program." Buddy
          them and celebrate the first correct run loudly.

PITFALL: "Why does a list need square brackets and commas -- why not just
list three variables next to each other?"
  Handle: "Ask them what would happen if we wanted a hundred moves instead
          of three -- would a hundred separate variable names actually be
          manageable? A list scales to any size with the exact same
          syntax; separate variables do not."
