🥇 Mission 1 of 3 — Break It (10 min)
Find the Bug: What If the Human Lies?
🐛 The setup

Your Tier C bot trusts the human completely. It types (low+high)//2, you answer "h", "l", or "c" — and it never checks whether your answers are even possible.

Try this on paper first: pretend the secret number is 37. Answer honestly for the first 3 guesses, then on the 4th guess — lie on purpose (say "h" when it should be "l").

🔍 Diagnose the failure mode

This is the same problem real AI systems face: noisy or adversarial input. A self-driving car's sensor can misread a stop sign; a spam filter's training label can be wrong on purpose.

What could eventually happen to low and high if enough lies stack up?

💡 Vocabulary: when a search range becomes impossible (low > high), that's called an invariant violation — a rule the algorithm assumed would always hold, broken.
🥈 Mission 2 of 3 — Fix & Scale It (20 min)
Build a Lie-Tolerant, Million-Number Bot

    🌍 Real World Connection
    Where adaptive, error-tolerant binary search shows up
    • Git bisect finds which commit introduced a bug among thousands, using the exact same halving strategy — and tolerates the developer occasionally marking a commit wrong.
    • Adaptive testing (like SHSAT/SAT computer-adaptive sections) narrows in on your true skill level by adjusting difficulty up or down after every answer — a "fuzzy" binary search on your ability.
    • Database index lookups use binary search across millions of sorted rows, and must handle corrupted or missing data without crashing — same contradiction-check idea.
