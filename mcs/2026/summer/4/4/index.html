Warm-Up · No Screens

How can one variable hold many choices?

Look & Predict

If our game choices are rock, paper, and scissors, should we store them in three separate variables or in one organized container? Why might one place be easier?

Turn & Talk

A partner picks a move. You say what beats it. Then switch. Which rule is easiest to remember? Which one could a computer check fastest?

Big idea: Lists help us group related items so our code can grab one, compare one, or loop through all of them.

Mini-Lesson Part 1

What is a Python list?

Real-world picture

A cubby shelf holds many folders in one place.
A playlist holds many songs in one order.
A list in Python holds many values in one variable.

Code picture

moves = ["rock", "paper", "scissors"]

print(moves)
print(moves[0])
print(moves[1])
print(moves[2])

Teacher prompt: If moves[0] is rock, what do you think moves[2] will be? Why do you think counting starts at 0 instead of 1?

Mini-Lesson Part 2

Indexing: how does Python find one item?

The list as numbered boxes

index 0
rock
index 1
paper
index 2
scissors

An index is the position number Python uses to reach inside the list and grab exactly one value.

Prediction challenge

What prints?

print(moves[1])

What if?

What mistake happens if we ask for an index that is not there?

Why useful?

A game menu, class roster, and score history can all be stored as lists.

Helpful misconception check: students often say “first item is 1.” Pause and have them point to the leftmost box and say, “computers begin at 0.”

Mini-Lesson Part 3

How does the computer choose a move?

New tool: random.choice()

import random

moves = ["rock", "paper", "scissors"]
computer_move = random.choice(moves)
print(computer_move)

Python looks inside the list and picks one item for us. We do not know which one in advance.

Ask before revealing

  • Could the computer pick something that is not in the list?
  • What must be true before random.choice(moves) can work?
  • Why is a list a perfect partner for random choice?
Prediction StopIf we add “lizard” and “spock,” the computer can pick from five items with the same one line of code.

Our game list

example 1
rock

A list is organized data. It is small, but it still stores examples the program can use.

Rule-based today

We write the rules ourselves: rock beats scissors, scissors beats paper, paper beats rock.

Modern AI later

Modern AI uses much bigger datasets and learns patterns from many examples instead of only following hand-written rules.

Guided Build · 20 min

Build the game, one thought at a time

1

Make the list

Type moves = ["rock", "paper", "scissors"]. Ask: why is this better than three separate variables?

2

Import random and pick for the computer

Use import random and computer = random.choice(moves).

3

Get the player’s move

Use input(). Ask students what exact words the player should type so comparisons will work.

4

Compare the two moves

Start with tie first. Then use elif branches to test each win or loss case.

5

Print both choices clearly

Students should see “You chose…” and “Computer chose…” before the result. This makes debugging easier.

Prediction pause: before running, have students whisper whether the code can ever say “You win!” when both choices match.

Model Code

Readable beginner version

import random

moves = ["rock", "paper", "scissors"]
player = input("Type rock, paper, or scissors: ").lower()
computer = random.choice(moves)

print("You chose:", player)
print("Computer chose:", computer)

if player == computer:
    print("Tie!")
elif player == "rock" and computer == "scissors":
    print("You win!")
elif player == "paper" and computer == "rock":
    print("You win!")
elif player == "scissors" and computer == "paper":
    print("You win!")
else:
    print("Computer wins!")

Ask: why do we call .lower() on the player input?

Ask: which line uses the list directly?

Ask: which branch catches all remaining lose cases?

Movement Break · 5 min

Stand up and become the program

Human Rock-Paper-Scissors Logic

IF teacher says “rock,” make a rock fist.
IF teacher says “paper,” open your hand flat.
IF teacher says “scissors,” make a V shape.
ELSE freeze like a confused computer and wait for a valid input.

Students feel the difference between valid inputs and invalid inputs before they return to the keyboard.

Open Build · 30–40 min

Three entry points, one game idea

Tier A · Foundations

Get one clean round working

Use the starter code. Focus on the list, one random computer choice, and correct win/lose/tie output.

Tier B · Core

Add score-keeping

Track player score and computer score across rounds. Print the score after each game.

Tier C · Accelerated

Add simple pattern memory

Keep a list of the player’s past moves and let the computer guess the most common move, then try to counter it.

★

Challenge Card · All Tiers

Extend the list to rock-paper-scissors-lizard-spock. Which part of the code becomes harder now? Which part stays easy because the list already exists?

Fast finisher move: ask students to refactor repeated win-checking into a helper function or a dictionary of rules.

Homework · 30 min

Two strong homework paths

Path 1 · Score across rounds

Add variables for player score and computer score. Play several rounds and print the updated score after each round.

This reinforces variables, loops, and repeated testing.

Path 2 · Extend the list

Add new items such as lizard and spock. Write the extra rules carefully and test every matchup you can think of.

This reinforces list thinking and precise logic.

Optional reflection line: “One thing lists made easier in my program was…”
