What Makes a Project "Shippable"?

Professionals ship software that works — even if it's small. A shippable Scratch project has these 3 qualities:

▶
1. It Starts
Click the green flag → something happens. Every time. No blank stage!
🎮
2. It Responds
Something reacts to YOU — a key press, mouse click, or sprite touching another.
🏁
3. It Ends (or Loops)
Either a game over message, OR a fun loop the player can watch/play forever.

Guided Build · Everyone Together · 20 min

Build the "Starter Shell" — Step by Step

We build this together. Every student ends this section with a working skeleton. Then you customize it your way.

1
Open Scratch & Pick Your Theme
Go to scratch.mit.edu → New Project. Choose a backdrop that fits your game theme (Space, City, Nature...).
2
Choose Your Main Sprite
Click the cat → delete or keep it. Pick a sprite that IS your player/character. Size it to about 60–80px.
3
Add the "When Flag Clicked" Hook
Drag When 🚩 clicked onto your script. Everything starts here.
4
Add a Forever Loop + Movement
Inside forever, add arrow-key conditionals OR mouse-follow. Test it now!
5
Add a Second Sprite + Interaction
Add something to catch, avoid, or collect. Use touching [sprite]? in a conditional.
6
Add a Score Variable
Make a variable → name it "Score" → set to 0 when flag clicked → change by 1 when you catch something.
7
Test the Minimum Shippable Version
Click green flag → does it start? Does it respond? Can you score? If yes — you have a shippable game!
Stuck? Check this first:
Is the code inside the forever loop? Did you click the green flag? Is the right sprite selected?

Predict the Output — Two Challenges

Challenge 2 — Spot the Bug
when 🚩 clicked
set [lives] to 3
if <touching [enemy]?> then
change [lives] by -1
The lives counter never goes down during play. What's the bug?
A) The variable "lives" is named wrong
B) It should say "change by -3" not "-1"
C) The if block is not inside a forever loop — it only checks once at the start

Debug Corner — When Things Go Wrong

Every Bug Has a Reason — Find It!

Bugs aren't failures — they're clues. Professional coders at Google, Apple, and MIT spend 50% of their time debugging. What does yours tell you?

Bug 1 — Nothing Happens on Flag Click
Why? Your code block is not connected to When 🚩 clicked
Fix: Drag the hat block to the very top of your script chain.
Bug 2 — If Block Only Works Once
Why? The conditional is outside the forever loop.
Fix: Nest the if INSIDE the forever loop so it checks continuously.
Bug 3 — Sprite Flies Off Screen
Why? No edge boundary check. Movement never stops.
Fix: Add if on edge, bounce from Motion blocks.
Bug 4 — Score Counts Too Fast
Why? The touching check fires every frame (30x per second!).
Fix: After touching, add wait 0.5 secs OR move the object to a new spot immediately.
The Debugging Mindset
1. What did I expect to happen?
2. What actually happened?
3. Which block controls that behavior?
4. Click that block alone to test it.
Pro Move
Right-click any block → "Help" — Scratch shows exactly what that block does. Use it!

Vocabulary · 6 Key Words

Words a Coder Knows — Can You Define Them?

Cover the definitions and try to explain each word to a partner. If you can teach it, you know it.

Algorithm
A precise, step-by-step set of instructions a computer follows. Your Scratch script IS an algorithm.
Event
Something that triggers code to run — like clicking the flag, pressing a key, or a sprite touching another.
Variable
A named container for a value that can change — like a score box that updates every time you catch something.
Conditional
Code that only runs when a condition is true — "IF touching apple THEN change score by 1." The basis of all decision-making.
Debug
Finding and fixing errors (bugs) in your code. The word "bug" comes from a real moth found in a computer in 1947!
Training Data (AI)
The examples AI learns from — like showing a robot 1,000 pictures of cats so it can recognize cats on its own.
