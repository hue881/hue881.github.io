✋ [Give a full 2 minutes — this is the conceptual heart of the day.]

💡 [THE BIG IDEA — this is the most important thing you'll say today]:
"This is the question that separates a beginner from a real thinker about AI.

Truth and fairness are not the same thing. A model can be trained on nothing
but true data and still be deeply unfair, because of what was left out, who
was over-counted, and whose story never made it into the dataset at all.
Remember this sentence: 'Bias often lives not in the lies you include, but in
the truths you leave out.'"

SLIDE 4 — VOCABULARY: SEVEN WORDS THAT UNLOCK TODAY
[Duration: ~5 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

"We have seven vocabulary words today — four about IDEAS, three about CODE.
Let's go through them carefully. These are the keys that unlock everything else."

"WORD 1: TRAINING DATA — the examples a model learns its patterns from.
  💡 This is the single most important concept in all of machine learning.
  A model is NOT programmed with rules the way old software was. Instead, it
  is SHOWN thousands of examples and figures out the patterns itself. The
  quality of those examples completely determines the quality of the output.
  Computer scientists have a famous phrase for this: 'garbage in, garbage
  out.' Bad data makes a bad model, no matter how clever the algorithm is."

"WORD 2: BIAS — a systematic error that unfairly skews predictions toward
one group.
  💡 Notice the word SYSTEMATIC. A random mistake isn't bias — bias is when
  the errors lean consistently in one direction, against the same group, over
  and over. That pattern of unfairness is what makes bias dangerous and hard
  to spot."

"WORD 3: CLASS IMBALANCE — when one category has far more examples than
another, like 900 cats versus 10 dogs.
  💡 This is the technical NAME for the cat-and-dog problem from our warm-up.
  Imbalanced data is one of the most common causes of bias in real AI systems,
  and you'll learn to measure it and fix it today."

"WORD 4: FAIRNESS METRIC — a NUMBER that measures how equally a model treats
different groups.
  💡 Here's the beautiful part. Fairness sounds fuzzy and philosophical — but
  computer scientists have found ways to turn it into actual numbers you can
  calculate. Once fairness is a number, you can measure it, track it, and
  improve it. We'll compute three of these numbers later today."

"Now the three CODE words:"

"WORD 5: TUPLE — an immutable pair, written like ('Alex', 'red').
  💡 A tuple is like a list, BUT it cannot be changed after you create it —
  that's what 'immutable' means. Why would we WANT something we can't change?
  Because a record — like a person's name paired with their team — shouldn't
  accidentally get edited halfway through your program. Tuples are perfect for
  storing fixed records, like a row in a spreadsheet."

"WORD 6: COUNTER — a special dictionary that counts things automatically.
  💡 Counter(['a','b','a']) instantly gives you {'a':2, 'b':1}. It's a tool
  built into Python that tallies how many times each item appears. We'll use
  it constantly today, because measuring bias usually starts with COUNTING."

"WORD 7: LIST COMPREHENSION — a one-line way to filter and transform a list,
like [x*2 for x in nums if x>0].
  💡 This is one of the most beloved features in all of Python. It lets you
  do in ONE line what used to take a whole loop. It reads almost like English:
  'give me x times two, for every x in nums, if x is positive.' By the end of
  today you'll be writing these like a pro."

SLIDE 5 — PYTHON DEEP DIVE: TUPLES & THE FOUR PATTERNS
[Duration: ~6 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[Have students open their Python editor / Replit / IDLE]

"Code time. Before we touch bias, we need four Python power-tools. Type these
in with me — we'll run each one and watch what happens."

"First, our data. Each player is a TUPLE — a name paired with a team color:"

  players = [
      ('Alex',  'red'),
      ('Blake', 'blue'),
      ('Casey', 'blue'),
      ('Devon', 'blue'),
      ('Emi',   'red'),
  ]

"Now four patterns. Pattern 1 — UNPACKING:"

  for name, team in players:
      print(name, "->", team)

💡 [EXPLAIN]:
"Watch what just happened. Each tuple has two parts. When we write
'for name, team in players', Python automatically UNPACKS each tuple — it puts
the first part into 'name' and the second into 'team.' No square brackets, no
[0] and [1]. This is cleaner, safer, and easier to read. Tuple unpacking is
one of the most Pythonic things you can do."

"Pattern 2 — ENUMERATE, which gives you the position AND the value:"

  for idx, (name, team) in enumerate(players):
      print(f"#{idx}: {name} is {team}")

💡 "enumerate() hands you a counter for free — idx is 0, then 1, then 2, and
so on. Before enumerate existed, programmers had to create a counter variable
and add one to it by hand every loop. This is the modern, mistake-proof way."

"Pattern 3 — LIST COMPREHENSION:"

  reds = [n for n, t in players if t == 'red']
  print(reds)   # → ['Alex', 'Emi']

💡 [SLOW DOWN — this is the star of the show]:
"Read this out loud as English: 'Give me n, for every name-and-team in
players, IF the team equals red.' One line did three jobs at once — it LOOPED,
it FILTERED, and it BUILT a new list. The general shape is always:
  [ what-you-want   for var in collection   if condition ]
Master this shape and you've mastered one of Python's superpowers."

"Pattern 4 — ZIP, which pairs two lists together:"

  names  = ['Alex', 'Blake', 'Casey']
  scores = [0.9, 0.7, 0.8]
  for name, score in zip(names, scores):
      print(f"{name}: {score:.0%}")

💡 "zip() is like a zipper on a jacket — it brings two separate rows together,
matching them up one-for-one: Alex with 0.9, Blake with 0.7, and so on. If the
two lists are different lengths, zip politely stops at the shorter one. And
notice {score:.0%} — that f-string trick turns 0.9 into a clean '90%'."

SLIDE 6 — CODE: THE COUNTER MEASURES IMBALANCE
[Duration: ~6 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

"Now we use the Counter to MEASURE bias. Type this below your players list:"

  from collections import Counter

  counts = Counter(team for _, team in players)
  print(counts)   # → Counter({'blue': 3, 'red': 2})

  total = sum(counts.values())
  ratio = counts['blue'] / counts['red']
  print(f"Ratio blue:red = {ratio:.2f}x")

  dominant = counts.most_common(1)[0]
  print(f"Robot prefers: {dominant[0]} ({dominant[1]}/{total})")

"Run it. You should see blue:red is 1.5x, and the robot prefers blue."

✋ [Wait — help with any import or indentation errors.]

💡 [LINE BY LINE — go slowly]:
"Let's read this like a story.

  Counter(team for _, team in players)
  → This counts every team color in ONE line. But look at that underscore.

  💡 THE UNDERSCORE: We wrote 'for _, team in players' — using _ instead of a
  real name. Why? Because we don't CARE about the player's name here; we only
  want the team. The underscore is a signal to other programmers that says: 'I
  know there's a value here, but I'm deliberately ignoring it.' It's a tiny bit
  of professional courtesy in your code — a note to the next human who reads it.

  total = sum(counts.values())
  → counts.values() gives us just the numbers (3 and 2). sum() adds them: 5.

  ratio = counts['blue'] / counts['red']
  → 3 divided by 2 = 1.5. This single number tells us HOW skewed the data is.
    A ratio of 1.0 would be perfectly balanced. The further from 1.0, the more
    imbalanced — and the more likely the model becomes biased.

  counts.most_common(1)[0]
  → most_common(1) finds the top item: ('blue', 3). The [0] grabs that pair
    out of the list. This is the category the robot will unfairly prefer,
    simply because it saw the most examples of it."

[Show the trace table on the slide]
"Watch HOW the Counter builds up, step by step:
  Start: {}              empty
  Alex,  red  → {red:1}
  Blake, blue → {red:1, blue:1}
  Casey, blue → {red:1, blue:2}
  Devon, blue → {red:1, blue:3}
  Emi,   red  → {red:2, blue:3}
  most_common(1) → [('blue', 3)]"

💡 [WHY COUNTER BEATS A NORMAL LOOP]:
"You could count this with a regular for-loop and a dictionary. But you'd have
to check 'is this key already here? If not, create it; if so, add one.' Counter
handles all of that automatically — and it never throws a KeyError on a new key.
It's also written in the C programming language under the hood, which makes it
about four times faster on big datasets. Less code, fewer bugs, more speed."

SLIDE 7 — BIAS IN THE WILD: THREE REAL DISASTERS
[Duration: ~6 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

"Everything we just coded is not a toy. These exact problems caused three
famous real-world failures. Let me tell you each story."

"DISASTER 1 — AMAZON'S HIRING AI (2014 to 2018).
Amazon built an AI to read résumés and rank job candidates. They trained it on
ten years of past résumés — but those résumés were 96% MALE. So the model
learned a pattern: 'successful applicants look male.' It started PENALIZING any
résumé containing the word 'women's' — like 'women's chess club captain' — and
it downgraded graduates of two all-women's colleges automatically. Amazon
eventually scrapped the whole system."

💡 "Connect it to our code: their training data had a ratio like 96-to-4. That
extreme class imbalance taught the model the wrong pattern. The model wasn't
sexist on purpose — it was a perfect mirror of biased historical data."

"DISASTER 2 — MIT FACE RECOGNITION (Joy Buolamwini, 2018).
A researcher named Joy Buolamwini tested three major commercial face-recognition
systems across different skin tones and genders. The error rate for
light-skinned men was 0.8%. For dark-skinned women, it was 34.7%. That's a
43-times gap in accuracy — for the SAME task."

💡 "This is the cat-and-dog problem from our warm-up, but with real human faces
and real consequences. The training data had far fewer dark-skinned faces, so
the model never learned them well. And remember — overall accuracy looked fine,
which is exactly why this went unnoticed until someone measured each group
SEPARATELY."

"DISASTER 3 — NHS MEDICAL AI (UK, 2021 audit).
A chest X-ray AI in Britain was trained mostly on patients of one ethnicity.
When used across all patients, accuracy DROPPED 11% for under-represented
groups — affecting triage decisions for about 140,000 scans a year."

💡 [LET THIS LAND]:
"Think about what that means. An 11% accuracy drop in MEDICINE isn't an
inconvenience — it could mean a missed diagnosis. This is no longer about
movie recommendations. This is life and death. That's the weight of getting
training data right."

SLIDE 9 — FIXING THE CODE: THREE BALANCING STRATEGIES
[Duration: ~6 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

"We've diagnosed the disease. Now the cure. There are three real strategies
data scientists use to balance imbalanced data. We'll code all three."

"STRATEGY 1 — COLLECT MORE DATA:"

  more = [('Finley', 'red'), ('Gray', 'red')]
  data = players + more
  counts = Counter(t for _, t in data)
  # → Counter({'red': 4, 'blue': 3})

💡 "This is the BEST option whenever it's possible. We simply gathered more
real examples of the rare group. Now the ratio is 4 to 3 — much closer to
balanced. Real data always beats fake data. The catch? Collecting more data
costs time and money, and sometimes it's just not available."

"STRATEGY 2 — UNDERSAMPLE the majority:"

  import random
  blues = [p for p in players if p[1] == 'blue']
  reds  = [p for p in players if p[1] == 'red']
  n = min(len(blues), len(reds))
  balanced = random.sample(blues, n) + random.sample(reds, n)

💡 "Here we THROW AWAY some of the majority group until both groups are equal.
Notice the list comprehensions filtering by p[1] — the team. We take only as
many blues as we have reds. It works, and it's quick — but it's WASTEFUL,
because we deleted good data. Use this when you have plenty of data to spare."

"STRATEGY 3 — WEIGHT BY CLASS (the clever one):"

  counts = Counter(t for _, t in players)
  total = sum(counts.values())
  weights = {label: total / count for label, count in counts.items()}
  # → {'blue': 1.67, 'red': 2.5}

💡 [THE STAR OF THE SLIDE — explain carefully]:
"This is the most elegant fix, and it uses a DICT COMPREHENSION — like a list
comprehension, but it builds a dictionary. Read it: 'for each label and count,
make an entry where the value is total divided by count.'

The key insight is the formula: weight = total / count. The RARER a group is,
the HIGHER its weight. Red appears only twice, so it gets weight 2.5 — meaning
we tell the model 'pay 2.5 times MORE attention to each red example.' Blue is
common, so it gets a lower weight of 1.67. We didn't add data, we didn't delete
data — we just told the model to CARE MORE about rare examples. This is exactly
what professional machine-learning systems do, using something called 'class
weights.'"

[Point to the before/after chart on the slide]
"And look at the result on the slide: after balancing, the accuracy gap between
groups shrinks from 38% all the way down to 3% — and neither group was
sacrificed to do it. That's a real fix."

SLIDE 10 — CODE: DICT COMPREHENSION & VISUAL BARS
[Duration: ~4 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

"Let's make our data SEE-able. Type this — we'll turn percentages into a bar
chart made of text:"

  from collections import Counter
  players = [('Alex','red'), ('Blake','blue'), ('Casey','blue'),
             ('Devon','blue'), ('Emi','red')]

  counts = Counter(t for _, t in players)
  total = sum(counts.values())

  pct = {label: round(count / total * 100, 1)
         for label, count in counts.items()}
  # → {'blue': 60.0, 'red': 40.0}

  for label, percent in pct.items():
      bar = "█" * int(percent // 10)
      print(f"{label:5s} {bar} {percent}%")

"Run it. You should see blue with a longer bar than red."

✋ [Wait for output — celebrate the little bar chart!]

💡 [EXPLAIN THE TRICKS]:
"Three neat things here:
  - The dict comprehension builds {blue: 60.0, red: 40.0} in one line.
  - '█' * int(percent // 10) repeats a block character. The // is FLOOR
    DIVISION — 60 // 10 gives 6, so blue gets 6 blocks. We just drew a chart
    with no graphics library at all, only text!
  - {label:5s} pads the label to 5 characters wide so the bars line up neatly.

These f-string formatting tricks — {:5s} for width, {:.2f} for decimals,
{:.0%} for percentages — are everywhere in professional Python."

✋ [Challenge them: "How would you also print the COUNT next to each percent?"
 Answer: add {counts[label]} into the f-string.]

SLIDE 11 — FAIRNESS METRICS: TURNING FAIRNESS INTO NUMBERS
[Duration: ~5 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

"Here's where it gets philosophically deep. We said fairness can be a NUMBER.
But it turns out there isn't just ONE definition of fair — there are several,
and they can disagree. Let's meet three."

"METRIC 1 — DEMOGRAPHIC PARITY:
The model gives positive outcomes to EQUAL PROPORTIONS of each group.
  💡 Example: if 30% of red players get picked for the team, then 30% of blue
  players should get picked too — regardless of skill. Sounds fair! But the
  warning on the slide is important: if the groups GENUINELY differ, forcing
  equal proportions can itself create unfairness."

"METRIC 2 — EQUAL OPPORTUNITY:
Among the people who SHOULD be selected, the model picks equal fractions from
each group.
  💡 This one is subtler and often fairer. It says: among everyone who truly
  deserves to be picked, make sure you don't favor one group's deserving people
  over another's. This is the metric most commonly used in hiring AI and
  medical triage, because it focuses on not failing people who genuinely
  qualify."

"METRIC 3 — CALIBRATION:
When the model says '70% confident,' it should be right 70% of the time — for
EVERY group equally.
  💡 This is about TRUST. If the model is over-confident for one group and
  under-confident for another, you can't trust its numbers the same way for
  everyone. This metric is critical for medical AI and risk-scoring tools,
  where a confidence number drives real decisions."

💡 [THE MIND-BLOWING PART — say this slowly and seriously]:
"Now here is one of the deepest results in all of AI ethics. In 2017, a
researcher named Alexandra Chouldechova proved something called an
IMPOSSIBILITY THEOREM. She showed mathematically that when two groups have
genuinely different base rates, you CANNOT satisfy all three of these fairness
definitions at the same time. It's not that we're not clever enough yet — it's
mathematically impossible. Every AI system that affects people must CHOOSE
which kind of fairness matters most. There is no perfectly fair algorithm.
There are only HONEST choices about which fairness you're prioritizing — and
that's a human decision, not a math one."

SLIDE 12 — BUILD CHALLENGE: MEASURE ACCURACY PER GROUP
[Duration: ~6 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

"Now YOU build. This is the big one — we measure a model's accuracy for EACH
group separately, exactly like the MIT researcher did. Here's the data: each
tuple is (name, true team, the model's prediction)."

  from collections import Counter

  results = [
      ('Alex',   'red',  'red'),    # correct ✓
      ('Blake',  'blue', 'blue'),   # correct ✓
      ('Casey',  'blue', 'blue'),   # correct ✓
      ('Devon',  'blue', 'red'),    # WRONG ✗
      ('Emi',    'red',  'blue'),   # WRONG ✗
      ('Finley', 'red',  'red'),    # correct ✓
  ]

"Your task — four steps:
  1) Count correct predictions per team.
  2) Count total examples per team.
  3) Print accuracy for each team.
  4) Flag any team where accuracy is below 70%."

✋ [Give 4-5 minutes. Circulate and help. Let them struggle a little first.]

[Then reveal one approach:]

  correct = {t: sum(1 for _, team, pred in results if team == t and pred == t)
             for t in ['red', 'blue']}
  total = Counter(t for _, t, _ in results)

  for t in total:
      acc = correct[t] / total[t] * 100
      flag = ' ⚠️ BIAS' if acc < 70 else ''
      print(f"{t}: {correct[t]}/{total[t]} = {acc:.1f}%{flag}")

💡 [WALK THROUGH IT]:
"Let's read the clever parts:

  sum(1 for _, team, pred in results if team == t and pred == t)
  → This counts correct predictions. For each result, IF the true team equals
    the prediction, we add a 1. Summing the 1s counts the correct ones. This
    little pattern — sum(1 for ... if ...) — is a Pythonic way to COUNT things
    that match a condition.

  Counter(t for _, t, _ in results)
  → Notice TWO underscores this time — we ignore both the name and the
    prediction, keeping only the true team. Counter tallies the totals.

  flag = ' ⚠️ BIAS' if acc < 70 else ''
  → This is a TERNARY expression — a one-line if/else. It adds a warning label
    only when accuracy falls below our 70% fairness threshold.

When you run it, you'll see one team flagged with a bias warning. THAT is
exactly the per-group measurement that exposed the MIT face-recognition gap.
You just built a fairness auditor."

💡 [STRETCH — for fast finishers]:
"If you finish early: compute PRECISION for red. Precision asks: out of all the
times the model PREDICTED 'red', how many were actually red? It's a different
question than accuracy, and it catches a different kind of mistake."

APPENDIX A — COMMON ERRORS & HOW TO FIX THEM
================================================================================

ERROR: ImportError / NameError: 'Counter' is not defined
  Cause: Forgot the import line.
  Fix:   Add 'from collections import Counter' at the TOP of your file.

ERROR: IndentationError
  Cause: Python uses indentation to define code blocks. One extra or missing
         space breaks everything.
  Fix:   Indent every line inside a loop or comprehension consistently —
         4 spaces. Don't mix tabs and spaces.

ERROR: ZeroDivisionError
  Cause: Computing a ratio when one group has zero members (e.g., no 'red').
  Fix:   Check that both groups exist before dividing, or guard with an if.

ERROR: KeyError
  Cause: Looking up counts['green'] when 'green' was never counted.
  Fix:   Use counts.get('green', 0) which safely returns 0 for missing keys.
         (Counter actually returns 0 automatically — but a plain dict won't.)

ERROR: SyntaxError in a comprehension
  Cause: Wrong order — putting the 'if' before the 'for'.
  Fix:   The order is ALWAYS [ expr  for var in iterable  if condition ].
